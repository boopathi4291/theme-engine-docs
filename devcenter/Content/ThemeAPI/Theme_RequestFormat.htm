Request Format

When contacting the API for data to perform actions, you will send a request to the VolusionREST API endpoints. The request will receive a response of a status code and a JSON response envelope of data.

Primarily,the Storefront API accepts and returns in JSON. All code samples used throughout this reference are JSON. The Apiary documentation provides live examples of method calls and response JSON for Raw HTTP, Curl, JavaScript, Node.js, Python, PHP, Ruby, C#, Visual Basic, Groovy, Objective-C, and Swift. The available Hosts include the mock server, bebugging proxy, and production.

Methods

Each API resource endpoint supports one or more methods for requests for data and Volusionfeatures. Requests to the API include the following methods:

Operation Description
GET

Retrieves data and items from the Volusion store. A successful request returns a status code of 200 with JSON displaying data and additional information as appropriate for pagination of results and possible warnings that occur.
POST

Creates a new object or item in the store. A successful request returns a status code of 201 and the JSON response of the new data. For example, a POST method against the User endpoint could create a new user account object in the store returning their ID.

PUT

Updates the data for an existing item in the store. A successful request returns a status code of 200 with a JSON response detailing the data. For example, a PUT operation for shipping information against the Cart endpoint will provide an updated shopping cart object with returned data for the shipping, cart totals, and other pertinent information.
DELETE

Deletes data for an existing item in the store. A successful request returns a 204 status without further data in the body of the JSON. For example, a DELETE method for an existing billing address returns a 204 and removes the billing address for that specific user account.

URI Structure

The request includes one of the methods and a URI structure that includes the API resource and any specific parameters.

The following is a sample of the URI structure:

METHOD https://{domain}/api/v1/{resource}/{resourceId}

Another format includes:

METHOD https://{domain}/api/v1/{resource}/{aspect}

The {domain}, you can enter one of the following:

  • Testing via Apiary, you can enter the mock server for Apiary: volusion.apiary-mock.com
  • Testing via a Volusion store, you can enter the sample store: www.samplestore.io

Depending on the request made and header, the URL structure may include additional parameters. For example, when getting a collection of reviews, you need to provide a parameter for the product code. All additional required and optional parameters are detailed per endpoint.

You can also use query methods to enhance the GET method. Example: 

GET https://{domain}/api/v1/products?page=2&pageSize=10

The following table details this structure in-depth:

Structure Description
{domain} The domain is your storefront domain. For the purposes of testing, you should use www.samplestore.io as the production domain. The domain volusion.apiary-mock.com is the Apiary mock server.
/api/v1/ All endpoints begin with this information as the location for calling that specific endpoint. The service endpoint includes the current version (v1). When subsequent release are provided, this version may change and need updating. Further information will be provided for future releases.
{resource} This signifies the resource endpoint for a specific set of data or objects in the system. Supported endpoints include products, categories, articles, navigation, config, components, reviews, carts, countries, users, login and orders. For details, see the list of resource endpoints.
{resourceId}

{aspect}

The resourceId sets the ID for a specific object to reference for the operation. Optionally, {aspect}can be used in place of resourceId. For example, currentcaptures current data for the referenced resource. For example, /users/current would retrieve the current shoppers without having to provide the specific ID.

Slug

To provide clean URLs for your content, a slug parameter is used for requests to the Products, Categories, and Articles. Slug is used in place of a name, code, or id to give an SEO friendly URL link for an object, generated from current values from these objects or added to the objects with new corresponding fields. Further details for the usage of this parameter is detailed per those endpoints. At a development level, a Dashify function strips out special characters from the value to ensure it is completely alphanumeric. The function replaces all spaces with dashes and lowercases all capitals. When created by script or entered and edited by a merchant, validation ensures the slug is unique across your storefront. If the value already is used and exists, you can change the entry. If the system is generating this value, a number is added. For example, the About Us article would be become about-us. If it exists, it would then be given about-us2 or prompt you to enter a different value.For another example, a URL normally for a category might display an id of 1477. Not the easiest content for a shopper to enter and save for a direct URL. If the category name was populated through the Dashify function into the slug, it could become home-decor. You may also wish to edit the slug URL from home-decor to homedecor.
